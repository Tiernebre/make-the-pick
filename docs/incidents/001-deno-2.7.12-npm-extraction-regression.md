# Incident 001: Deno 2.7.12 npm extraction regression blocked deploys

- **Date:** 2026-04-09 → 2026-04-10
- **Duration of broken `main`:** ~13 hours (last green deploy 2026-04-09 19:35
  UTC → fix merged 2026-04-10 07:45 UTC)
- **User impact:** No production outage — the live site kept running on the
  previous image. Impact was developer-facing: every push to `main` failed CI,
  so no new changes could be deployed to https://makethepick.gg during the
  window.
- **Severity:** SEV-3 (deploy pipeline blocked, production stable)

## Summary

The CI `Test` job started failing on every push to `main` with:

```
Error: Cannot find module '/home/runner/.../node_modules/.deno/
react-remove-scroll@2.7.2/node_modules/react-remove-scroll-bar/dist/es5/
constants.js'. Please verify that the package.json has a valid "main" entry
```

It reproduced on Linux CI only — local macOS runs were green. Five separate fix
attempts against `vitest.config.ts` (stubbing `react-remove-scroll`, inlining
deps, `vi.mock`, `resolve.alias`, ESM `__dirname`) all failed because **none of
them were targeting the real cause**.

The real cause: the CI workflow pinned Deno with `deno-version: v2.x`, which
floated up from `2.7.11` (last green) to `2.7.12` overnight. Deno 2.7.12
regressed how it materialises npm packages into the isolated
`node_modules/.deno/<pkg>/node_modules/` layout, and the
`react-remove-scroll-bar` package (a transitive dep of `@mantine/core` via
`react-remove-scroll`) ended up with a missing `dist/es5/constants.js` at
resolve time. macOS did not reproduce because the developer's machine still had
`2.7.11` cached in its Deno install dir.

## Root cause

Single line, in `.github/workflows/ci.yml`:

```yaml
- uses: denoland/setup-deno@v2
  with:
    deno-version: v2.x # <- floating, not pinned
```

Floating `v2.x` gave upstream Deno releases direct write access to the CI
environment. When Deno 2.7.12 shipped a regression in its npm package
materialisation, every CI run broke the next morning without any change to this
repo.

## What went well

- **Production stayed up.** The deploy pipeline fails _before_ the Droplet
  pull/restart step, so the prior good image kept serving traffic. The
  blast-radius was limited to "no new deploys".
- **Control probe killed a dead-end investigation fast.** After the first two
  bisect probes both failed, running CI at the last-known-green commit
  (`e95d63e9`) was the single highest-signal experiment available — it ruled out
  ~60 commits in ~3 minutes and pivoted the investigation from "find the bad
  commit" to "find the bad environment variable".
- **Making `continue-on-error` PR-conditional**
  (`github.event_name !=
  'pull_request'`) let deploys proceed while still
  giving bisect PRs an honest red/green signal. One workflow, two behaviours.
- **CI logs had the answer.** Grepping setup-deno output across runs showed the
  version jump immediately once the investigation was pointed at the
  environment.

## What went badly

- **Floating toolchain version in CI.** `deno-version: v2.x` was a latent
  supply-chain hazard: it made "what happens on main" depend on whatever Deno
  released in the last 24h. This is the same class of bug as `node:latest` or an
  unpinned base image — the fix is always "pin, then intentionally bump".
- **Five fix attempts before stopping to diagnose.** The previous session kept
  iterating on `vitest.config.ts` under the assumption that the failure was a
  vitest/ESM resolution bug. The error path
  (`.deno/react-remove-scroll@2.7.2/node_modules/...`) was a Deno layout detail,
  not a vitest one — that should have been a tell. Lesson: if fix N doesn't
  work, fix N+1 should not be "try another config tweak" — it should be "form a
  new hypothesis based on what the error _is actually describing_".
- **"Works on my Mac" was not interrogated early enough.** A Linux-only,
  macOS-green failure against an identical lockfile is a _very_ narrow set of
  possibilities — toolchain version, runner OS, or file system case sensitivity.
  Leading with "what's different between the two environments?" would have found
  this in one step instead of five.
- **No CI on PRs.** CI ran only via `workflow_call` from `deploy.yml`, which
  only triggers on push to `main`. That meant every experiment had to go
  directly through `main` (or a bisect PR + workflow edit). No pre-merge gate
  existed — broken code reached `main` before anyone saw it fail. This is now
  fixed by the `pull_request` trigger added in `3f5b684`.
- **No incident log existed.** This is the first document in `docs/incidents/` —
  prior incidents weren't written up, so there's no pattern library of "things
  that have broken and why" to consult when something looks familiar.

## Action items

- [x] **Pin Deno in CI.** Done in `7b50650`: `deno-version: v2.7.11` in every
      job.
- [x] **Run CI on pull requests.** Done in `3f5b684`: `on: pull_request` with
      `branches: [main]`, so PRs get full lint/format/test/docker-smoke/e2e.
- [x] **Create `docs/incidents/`.** This document.
- [ ] **Pin other floating tool versions.** Audit the repo for `latest`, `@v2`,
      `v2.x`, or similar floating refs on anything that runs in CI or in
      production containers (base images, actions, npm/deno deps pulled at
      runtime). Replace with pinned versions + a deliberate update workflow.
- [ ] **Add a "Deno toolchain bump" ritual.** When unpinning or bumping Deno, do
      it in a dedicated PR so CI runs against the new version in isolation and
      any regressions are obviously attributable.
- [ ] **File upstream.** Reproduce Deno 2.7.12 npm extraction regression against
      a minimal repo with `@mantine/core` + vitest and open an issue at
      denoland/deno. Unpin once a fixed release ships.
- [ ] **Write a "debugging checklist" note.** First questions when CI fails but
      local passes: (1) what version of the toolchain is each environment
      running? (2) is the lockfile identical? (3) is the OS/arch different?
      Before touching config files.
